# Anki Repository Agent Instructions

This repository stores Anki card designs, note schemas, and note entries. As an AI agent assisting in this repository, you must strictly adhere to the following rules, workflows, and terminology.

## 1. Terminology & Communication
- **Strict Anki Terminology:** You must use Anki's official terminology at all times. 
  - **Note Type**: The schema defining a set of fields and which cards to generate.
  - **Card Type**: The blueprint (templates) defining the question/answer layout.
  - **Template**: The actual Front and Back HTML/CSS.
  - **Field**: A specific data slot in a Note (e.g., `Word`, `Translation`).
  - **Note**: A single data entry containing content for the fields.
  - **Card**: The actual flashcard generated by Anki.
  - **Deck** and **Collection**.
- **Ambiguity Check:** If the user uses non-standard words (e.g., "flashcard layout" instead of "Template", or "card template" instead of "Note Type"), you must pause and clarify before writing code or modifying files.
- **Special Fields:** When referring to Anki's native special fields (like tags), use standard Anki syntax (e.g., `{{Tags}}`).

## 2. Repository Structure
The repository follows a nested structure where Card Types and Note Entries belong to their parent Note Type. You must maintain this exact hierarchy:

```text
/note-types/
  {NoteTypeName}/
    note-type.md             ← Schema and ID strategy definition
    /entries/
      {NoteTypeName}.csv     ← Note entries for import
    /card-types/
      {CardTypeName}/
        front.html           ← Front template
        back.html            ← Back template
        style.css            ← Styling specific to this Card Type
        design.md            ← Card Type design rationale
```

## 3. Note Type Architecture
Every Note Type must have a `note-type.md` document that serves as the source of truth for its schema. The document must define:
- **Fields:** The exact name, purpose, and order of fields.
- **ID Strategy for Imports:** Explicitly define how Anki will match existing notes (e.g., "User-defined custom ID in the first field" or "Anki GUID column via `#guid column`").
- **Card Types:** A list of Card Types this Note Type generates.
- **Target Deck(s):** Where the resulting cards should be placed.

## 4. Card Type Architecture & Styling
- **Scoping:** Styling (`style.css`) is defined **per Card Type**. Do not assume a global stylesheet exists unless explicitly instructed.
- **Source Files:** The templates must be split into `front.html`, `back.html`, and `style.css`. 
- **Design Document:** Each Card Type requires a `design.md` explaining the learning intention, field mappings (e.g., `{{FieldName}}`), and any conditional logic (e.g., `{{#FieldName}}` / `{{^FieldName}}`).

## 5. Note Entries & Idempotent Import Workflow
Entries are stored as UTF-8 CSV files designed for repeatable imports.
- **File Headers:** Every CSV must include Anki file headers mapping to the Note Type's ID strategy:
  ```text
  #notetype:YourNoteTypeName
  #deck:YourDeckName
  #separator:Comma
  #html:true
  #tags column:5
  ```
- **Repeatable Imports:** Import must always be treated as an "Update existing notes" operation. You must structure the CSV to respect the ID Strategy defined in `note-type.md`.
- **Deletion Workflow (Tagging Workaround):** 
  - Anki's text import does not delete notes when rows are removed from a CSV.
  - To delete a note, **do not delete the row** from the CSV immediately.
  - Instead, add a deletion tag (e.g., `REPO_DELETE`) to the note's tags column.
  - Inform the user to import the CSV, search for `tag:REPO_DELETE` in Anki to manually delete the notes, and only *afterward* can the row be safely purged from the repository.

## 6. Active Validation & Linting
To minimize errors, you must actively validate the consistency of the repository whenever creating or modifying files:
1. **Field Validation:** Ensure every field referenced in a Card Type's `front.html` or `back.html` (e.g., `{{Concept}}`) is explicitly declared in the parent `note-type.md`.
2. **CSV Column Alignment:** Verify that the number of columns and their implicit order in the `.csv` files exactly match the Field list defined in `note-type.md`. 
3. **ID Strategy Compliance:** If the `note-type.md` mandates a first-field custom ID, ensure the first column of the CSV is never empty and contains stable identifiers. If it mandates a GUID, ensure the `#guid column` header is present.
4. **Error Reporting:** If the user requests a change that breaks this consistency (e.g., adding data to a CSV for a field that doesn't exist in the Note Type), you must block the action, explain the discrepancy, and ask how to proceed.
